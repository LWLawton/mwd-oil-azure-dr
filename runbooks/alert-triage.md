## Purpose

This runbook defines the standard triage process for all security alerts received by the MWD Oil Co. SOC, whether generated by Microsoft Sentinel, Defender for Cloud, endpoint protection, OT monitoring tools, or vendor-reported events.

---

## Alert Sources

| Source | Tool | Priority Weight |
|--------|------|----------------|
| Microsoft Sentinel | SIEM | High |
| Defender for Cloud | CSPM/EDR | High |
| OT/ICS monitoring | Site historians / NSG flow logs | Critical |
| Phishing reports | End user email / helpdesk | Medium |
| Vendor alerts | Email / vendor portal | Medium |
| Azure Health alerts | Azure Monitor | High |

---

## Step 1 — Receive and Acknowledge

- [ ] Alert received in Sentinel, email, or ticketing system
- [ ] Open or create a security ticket immediately
- [ ] Record: alert name, time of detection, source system, assigned analyst
- [ ] Set initial severity (P1–P4) based on the severity matrix below
- [ ] Acknowledge alert in Sentinel (prevents duplicate triage)

**Severity Matrix:**

| Severity | Criteria | Response SLA |
|----------|----------|-------------|
| P1 — Critical | Active compromise, OT impact, ransomware indicator, DR criteria met | Immediate / 15 min |
| P2 — High | Confirmed malicious activity, privileged account anomaly | 30 minutes |
| P3 — Medium | Suspicious activity, policy violation, failed authentication spike | 2 hours |
| P4 — Low | Informational, low-confidence, tuning candidate | Next business day |

---

## Step 2 — Initial Enrichment

Before escalating or taking action, enrich the alert with available context.

- [ ] **Who:** What user, service principal, or system generated the alert?
- [ ] **What:** What action or behavior triggered the alert? What is the raw event?
- [ ] **Where:** What resource, subnet, or site is involved? Is it IT or OT?
- [ ] **When:** When did the behavior begin? Is this an ongoing activity?
- [ ] **Why (hypothesis):** What is the most likely explanation — attack, misconfiguration, or false positive?

**Useful Sentinel KQL — Quick Enrichment:**

```kusto
// Recent sign-in activity for a user
SigninLogs
| where UserPrincipalName == "REPLACE_WITH_UPN"
| where TimeGenerated > ago(24h)
| project TimeGenerated, IPAddress, Location, AppDisplayName, ResultType

// Recent activity from a specific IP
CommonSecurityLog
| where TimeGenerated > ago(1h)
| where SourceIP == "REPLACE_WITH_IP"
| project TimeGenerated, Activity, DeviceAction, DestinationIP, DestinationPort

// VM activity summary
Heartbeat
| where Computer == "REPLACE_WITH_VM_NAME"
| where TimeGenerated > ago(2h)
| summarize LastSeen = max(TimeGenerated), count() by Computer, OSType
```

---

## Step 3 — Classify the Alert

Based on enrichment, classify the alert into one of the following:

| Classification | Action |
|---------------|--------|
| **True Positive — Active** | Escalate to P1/P2, begin containment, notify SOC Lead |
| **True Positive — Resolved** | Document, close ticket, submit tuning recommendation |
| **False Positive** | Close ticket with detailed rationale, submit tuning request |
| **Benign True Positive** | Expected behavior that matched a rule (e.g., admin task) — close with note |
| **Undetermined** | Escalate to SOC Lead for second opinion — do not close |

---

## Step 4 — OT / ICS Alert Special Handling

If the alert involves any OT-related asset:

- [ ] **Pause** — Do not take automated or remote action without OT team coordination
- [ ] Notify the OT site supervisor (Drill Site A or B) before touching anything
- [ ] Confirm whether the alert may affect physical operations or safety systems
- [ ] If ransomware or wiper is suspected in OT DMZ: isolate at NSG level, do NOT power off VMs without OT approval
- [ ] Escalate to P1 regardless of tool-assigned severity
- [ ] See `ransomware-containment.md` if encryption activity is confirmed

---

## Step 5 — Escalation

Escalate if any of the following are true:
- Alert involves privileged accounts or Entra ID Global Admin
- Alert involves OT, IoT, or drill site assets
- Alert correlates with 2+ other Sentinel incidents in the past 24 hours
- You cannot explain the activity after enrichment
- Alert meets any DR declaration criteria (see DR_RUNBOOK.md)

**Escalation Path:**
- SOC Analyst → SOC Lead (Slack/phone)
- SOC Lead → CISO (if P1 or confirmed compromise)
- CISO → Incident Commander (if DR is declared)

---

## Step 6 — Document and Close

For every triaged alert — even false positives:

- [ ] Document the classification and rationale in the ticket
- [ ] Record all enrichment queries and results
- [ ] Record disposition (closed, escalated, tuning submitted)
- [ ] If false positive: submit tuning request to improve the analytics rule
- [ ] Update ticket status and add closure notes

---

## Triage Quality Standards

The MWD Oil Co. SOC measures triage quality on:
- Time to acknowledge (target: <5 min for P1, <15 min for P2)
- False positive rate per rule (review rules >30% FP rate)
- Escalation accuracy (over- and under-escalation both tracked)
- Documentation completeness (ticket audited monthly)
